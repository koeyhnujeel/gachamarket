# Gachamarket MVP Core Platform Design

작성일: 2026-03-29
상태: 승인됨
범위: 스포츠 예측 서비스 MVP의 코어 플랫폼 설계

## 1. 목적

이 문서는 Gachamarket MVP의 제품 범위, 도메인 경계, 운영 흐름, 예외 처리, 프론트엔드 구조, 테스트 전략을 고정한다. 구현 단계에서는 이 문서를 기준으로 모듈 경계와 핵심 정책을 유지한다.

## 2. 제품 목표

- 스포츠 경기 결과 예측을 포인트 기반으로 즐길 수 있는 웹 서비스를 만든다.
- 포인트는 절대 현금으로 환전되지 않는다.
- 단순 베팅 화면이 아니라 댓글, 리더보드, 칭호를 통해 커뮤니티성과 성취감을 함께 제공한다.
- MVP 단계에서는 운영 복잡도를 통제하면서도, 이후 정치/기타 카테고리로 확장 가능한 구조를 준비한다.

## 3. MVP 범위

- 카테고리: 스포츠만 지원
- 지원 종목:
  - 축구: 해외 5대 리그, K리그1, 단기 국가대표 경기
  - 야구: MLB, KBO
  - 농구: NBA
- 마켓 종류: 경기 결과만 지원
  - 축구: 승/무/패
  - 야구, 농구: 승/패
- 회원 기능:
  - Google 로그인
  - 서비스 내부 자동 닉네임 생성
  - 닉네임 변경
  - 보유 칭호 중 1개 활성화
- 예측 기능:
  - 포인트 베팅
  - 같은 결과에 한해 추가 베팅 가능
  - 반대 결과로 변경 불가
  - 취소 불가
- 커뮤니티 기능:
  - 이벤트별 댓글
  - 댓글 좋아요
  - 대댓글 depth 1
- 경쟁 기능:
  - 전체 리더보드
  - 종목별 리더보드
  - 포인트 칭호
  - 리그별 숙련 칭호
  - 상위 합성 칭호
- 운영 기능:
  - 경기 수집 데이터 승인 큐
  - 이벤트 공개 승인
  - 수동 오즈 입력
  - 결과 재수집
  - 정산 재시도
  - 이벤트 중단 및 환불

## 4. 비목표

- 정치/기타 카테고리 지원
- 사용자 생성 예측 이벤트
- 실시간 가격 변동형 마켓
- 오버/언더, 핸디캡, 선수별 특수 마켓
- 기간제/시즌제 칭호
- 실시간 스트리밍 UI
- 별도 관리자 앱
- 모바일 앱

## 5. 확정 정책

- 예측 이벤트 원천 데이터는 무료 API 우선, 부족하면 크롤링으로 수집한다.
- 수집된 경기는 자동으로 시스템에 들어오지만, 공개는 운영자 승인 후 진행한다.
- 배당은 운영자 승인 시점에 확정하고, 공개 후 경기 시작 5분 전까지 고정한다.
- 외부 오즈가 없으면 운영자가 수동으로 오즈를 입력할 수 있다.
- 수동 오즈 입력 시 입력 근거 메모와 감사 로그를 반드시 남긴다.
- 경기 시작 5분 전 자동 마감한다.
- 경기 결과는 외부 데이터로 자동 수집하고, 이상이 없으면 자동 정산한다.
- 데이터가 모순되거나 결과가 불명확하면 자동 정산하지 않고 검토 상태로 멈춘다.
- 비로그인 사용자는 읽기만 가능하다.
- 로그인 사용자는 베팅, 댓글, 좋아요, 닉네임 변경, 무료 충전, 칭호 활성화를 할 수 있다.
- 리더보드 대표 기준은 현재 보유 포인트다.
- 포인트 지급 규칙:
  - 가입 시 1,000P 지급
  - 포인트가 0일 때만 하루 1회 100P 무료 충전 가능
- 닉네임 변경 규칙:
  - 최초 1회 무료
  - 이후 100P 소모

## 6. 아키텍처 방향

### 6.1 백엔드

- Java + Spring Boot
- Spring Modulith
- 헥사고날 아키텍처
- 도메인 모델 패턴
- 하나의 모듈형 모놀리스로 시작한다.
- 외부 시스템 연동, 배치, 승인 백오피스, 사용자 API를 하나의 애플리케이션 안에서 모듈로 분리한다.

### 6.2 프론트엔드

- Next.js 기반 웹 애플리케이션
- 사용자 화면과 관리자 화면을 같은 애플리케이션 안에서 제공한다.
- PC와 모바일을 모두 고려한 반응형 웹 설계
- 읽기 중심 화면은 서버 렌더링 우선
- 쓰기 중심 인터랙션은 클라이언트 섬 형태로 분리

### 6.3 인증

- Google 로그인(OAuth2/OIDC 기반)
- 인증 처리는 백엔드가 전담한다.
- Google 요청 스코프는 `openid`, `email`만 사용한다.
- 세션은 HttpOnly Secure Cookie 중심으로 유지한다.
- 프론트엔드는 Bearer 토큰을 직접 저장하지 않는다.

## 7. 모듈 분해

### 7.1 Identity

책임:
- Google 로그인 처리
- 회원 등록
- 닉네임 생성/변경
- 사용자 권한
- 활성 칭호 선택

핵심 모델:
- Member
- AuthIdentity
- NicknameChangePolicy

### 7.2 Category Catalog

책임:
- 카테고리 계층 관리
- 리더보드/칭호 집계 대상 계층 관리
- 화면 노출용 카테고리 트리 제공

핵심 모델:
- Category
- CategoryPath

초기 계층 예시:
- 스포츠
- 스포츠 > 축구
- 스포츠 > 축구 > EPL
- 스포츠 > 축구 > K리그1
- 스포츠 > 야구 > MLB
- 스포츠 > 농구 > NBA

### 7.3 Source Intake

책임:
- 외부 데이터 소스 연동
- 무료 API 호출
- 크롤링 보완
- 외부 포맷을 서비스 표준 모델로 정규화
- 원천 데이터 변경 감지

핵심 모델:
- SourceProvider
- SourceItem
- SourceItemMapping
- SourceSyncLog

MVP 구체화:
- MVP의 SourceItem은 경기 Fixture다.
- 미래 확장 시 정치/기타 카테고리용 SourceItem 타입을 추가할 수 있다.

### 7.4 Odds

책임:
- 외부 오즈 수집
- 오즈 스냅샷 저장
- 서비스 오즈 계산
- 수동 오즈 입력 감사 기록

핵심 모델:
- OddsSnapshot
- ConsensusOdds
- ManualOddsEntry

배당 계산 규칙:
- 외부 소스에서 outcome별 decimal odds를 수집한다.
- outcome별 implied probability를 계산한다.
- 이벤트 단위 overround를 정규화해 소스별 확률 합을 1.0으로 맞춘다.
- 같은 outcome의 정규화 확률들에 대해 중앙값을 사용한다.
- 중앙값 확률을 다시 decimal odds로 변환한다.
- 화면 표시는 소수점 둘째 자리까지 반올림한다.
- 승인 시점에 계산된 odds를 Prediction Event에 복사해 고정한다.

수동 오즈 입력 규칙:
- 모든 outcome 오즈를 함께 입력해야 한다.
- 입력자, 입력 시각, 근거 메모를 남긴다.
- 승인 후에는 수정할 수 없다.

### 7.5 Prediction Event

책임:
- 서비스 안에서 공개되는 예측 이벤트 관리
- 공개 여부, 마감 시각, 결과 반영 상태 관리
- 베팅 대상 단위 유지

핵심 모델:
- PredictionEvent
- PredictionOutcome
- EventLifecycle

PredictionEvent와 SourceItem은 분리한다.
- SourceItem은 외부 원천 데이터다.
- PredictionEvent는 서비스 안에서 실제로 노출되고 베팅 가능한 객체다.
- 한 SourceItem에 대해 반드시 하나의 PredictionEvent만 만들 필요는 없지만, MVP에서는 1:1로 유지한다.

상태:
- Imported
- PendingApproval
- Published
- Locked
- AwaitingResult
- ReviewRequired
- Suspended
- Settled
- Refunded

### 7.6 Betting & Wallet

책임:
- 포인트 잔액 관리
- 베팅 접수
- 추가 베팅
- 무료 충전
- 포인트 변동 원장 유지

핵심 모델:
- Wallet
- BetPosition
- BetSelection
- PointLedgerEntry
- FreeRechargeRecord

베팅 규칙:
- 사용자는 이벤트당 하나의 결과만 선택할 수 있다.
- 같은 결과에는 추가 베팅 가능하다.
- 반대 결과로 변경할 수 없다.
- 마감 후 베팅할 수 없다.
- 베팅 취소는 지원하지 않는다.
- 베팅 시점에 필요한 포인트가 부족하면 실패한다.

무료 충전 규칙:
- 현재 보유 포인트가 0이어야 한다.
- 당일 무료 충전을 아직 사용하지 않았어야 한다.
- 충전 성공 시 100P를 지급한다.

포인트 원장 규칙:
- 모든 포인트 증감은 PointLedgerEntry로 남긴다.
- 현재 잔액은 원장에서 계산 가능한 값이어야 한다.
- 중복 정산 방지를 위해 정산 트랜잭션 키를 둔다.

### 7.7 Settlement

책임:
- 결과 반영
- 승패 판정
- 배당 기반 포인트 정산
- 환불 처리
- 재시도 처리

핵심 모델:
- SettlementJob
- SettlementResult
- RefundDecision

정산 규칙:
- 결과 소스가 정상이고 이벤트가 Locked 또는 AwaitingResult 상태여야 한다.
- 베팅 시점에 stake는 먼저 차감한다.
- 승리 포지션 정산 시 `stake x fixed_odds`를 지급한다.
- 패배 포지션은 추가 지급 없이 종료한다.
- 환불 시 차감했던 stake만 그대로 돌려준다.
- 취소/연기/무효 경기면 환불한다.
- 이미 정산된 이벤트는 다시 정산하지 않는다.

자동 정산 중단 조건:
- 결과가 수집되지 않음
- 복수 소스 간 결과 충돌
- SourceItem과 PredictionEvent 매핑 이상
- 정산 대상 outcome을 판정할 수 없음

이 경우 이벤트는 ReviewRequired 또는 Suspended로 이동한다.

### 7.8 Title

책임:
- 칭호 규칙 관리
- 칭호 획득 판정
- 상위 합성 칭호 판정
- 사용자 보유 칭호 관리

핵심 모델:
- TitleRule
- Title
- UserTitle
- TitleProgress

규칙:
- 사용자는 보유 칭호 중 1개만 활성화할 수 있다.
- 포인트 칭호는 고정 규칙이다.
- 리그/카테고리 칭호는 운영자 설정 가능 규칙이다.
- 합성 칭호는 하위 칭호 보유 여부로 판정한다.
- 한 번 획득한 칭호는 회수하지 않는다.

MVP 기본 정책:
- 포인트 칭호는 사용자가 도달한 역대 최고 보유 포인트 구간 기준이다.
- 리그 칭호는 해당 리그 Prediction Event 적중 횟수 기준이다.
- 상위 칭호는 지정된 하위 칭호를 모두 보유하면 획득한다.

초기 기본 규칙:
- 리그 숙련 칭호 기본 조건은 해당 리그 적중 50회다.
- 축구 상위 합성 칭호는 지정된 축구 리그 칭호를 모두 획득하면 부여한다.
- 포인트 칭호 초기 구간은 1,000 / 5,000 / 10,000 / 25,000 / 50,000 / 100,000으로 시작한다.

### 7.9 Community

책임:
- 댓글
- 대댓글
- 좋아요
- 삭제/수정 정책

핵심 모델:
- Comment
- CommentLike

규칙:
- 댓글은 Prediction Event 단위로 작성한다.
- depth 1 대댓글만 허용한다.
- 사용자는 자신의 댓글만 수정/삭제할 수 있다.
- 삭제는 soft delete로 처리한다.
- 좋아요는 사용자당 댓글당 1회만 허용한다.
- 비로그인 사용자는 열람만 가능하다.

### 7.10 Leaderboard

책임:
- 전체 리더보드
- 종목별 리더보드
- 순위 계산 및 조회 최적화

핵심 모델:
- LeaderboardEntry
- LeaderboardProjection

규칙:
- 전체 리더보드는 현재 보유 포인트 내림차순이다.
- 종목별 리더보드는 전역 Wallet과 별도로 계산한 종목별 누적 순포인트 projection 내림차순이다.
- 전체 리더보드와 종목별 리더보드를 모두 제공한다.
- 동점자는 같은 순위를 공유한다.
- 종목별 projection은 해당 종목에서 발생한 stake 차감, 정산 지급, 환불만 반영한다.

### 7.11 Admin

책임:
- 승인 큐 조회
- 이벤트 승인/보류/반려
- 수동 오즈 입력
- 결과 재수집
- 정산 재시도
- 이벤트 중단 및 환불
- 리그/카테고리 칭호 규칙 수정

핵심 모델:
- AdminAuditLog
- AdminAction

관리자 권한:
- MVP는 단일 `ADMIN` 역할로 시작한다.

## 8. 핵심 사용자 흐름

### 8.1 로그인

1. 사용자가 Google 로그인 버튼을 클릭한다.
2. 백엔드가 Google 인증을 처리한다.
3. 식별 가능한 최소 정보로 회원을 찾거나 생성한다.
4. 신규 회원이면 자동 닉네임과 1,000P를 지급한다.
5. 세션 쿠키를 발급한다.

### 8.2 경기 수집과 게시

1. Source Intake가 경기 일정과 기본 정보를 수집한다.
2. Odds 모듈이 가능한 경우 외부 오즈를 수집한다.
3. 경기 단위로 PendingApproval 이벤트를 만든다.
4. 운영자는 승인 큐에서 이벤트를 검토한다.
5. 승인 시점에 오즈를 확정하고 Published 상태로 전환한다.
6. 오즈가 없으면 운영자가 수동 오즈를 입력한 뒤 승인할 수 있다.

### 8.3 베팅

1. 사용자가 Published 이벤트 상세 화면에 진입한다.
2. 마감 전인지 확인한다.
3. 잔액과 선택 결과를 확인한다.
4. 기존 포지션이 없으면 새 BetPosition을 만든다.
5. 기존 포지션이 있고 같은 결과면 포인트를 누적한다.
6. 기존 포지션이 있는데 반대 결과면 요청을 거절한다.
7. 포인트 원장과 잔액을 갱신한다.

### 8.4 마감과 정산

1. 경기 시작 5분 전에 이벤트를 Locked로 변경한다.
2. 결과 수집 성공 시 AwaitingResult에서 정산 가능 상태로 이동한다.
3. Settlement가 결과를 기준으로 지급/차감을 계산한다.
4. PointLedgerEntry를 기록한다.
5. Event를 Settled로 변경한다.
6. Title과 Leaderboard 갱신 이벤트를 발행한다.

### 8.5 무료 충전

1. 사용자가 무료 충전을 요청한다.
2. 잔액이 0인지 확인한다.
3. 당일 사용 기록이 없는지 확인한다.
4. 조건을 만족하면 100P 지급 후 사용 기록을 남긴다.

## 9. 운영 안전장치와 예외 처리

원칙:
- 데이터가 이상하면 자동으로 강행하지 않고 멈춘다.

예외 처리 규칙:
- 오즈 없음:
  - 운영자 수동 입력으로 보완 가능
  - 근거 메모와 감사 로그 필수
- 경기 시간 변경:
  - 재검토 큐로 보낸다.
  - 아직 Published면 운영자 재확인을 요구한다.
- 결과 미수집:
  - AwaitingResult 유지
  - 자동 정산하지 않음
- 복수 소스 결과 불일치:
  - ReviewRequired 이동
  - 자동 정산하지 않음
- 경기 취소/연기:
  - Suspended 이동 후 환불 처리
- 정산 중 오류:
  - 재시도 큐에 적재
  - 중복 정산 방지 키로 재실행 안전성 보장

백오피스 최소 액션:
- 승인
- 보류
- 반려
- 수동 오즈 입력
- 결과 재수집
- 정산 재시도
- 중단 및 환불

## 10. 프론트엔드 설계

### 10.1 사용자 화면

- 홈
- 카테고리/리그 탐색
- 이벤트 상세
- 리더보드
- 프로필

이벤트 상세 핵심 요소:
- 리그/경기 정보
- 배당
- 마감 카운트다운
- 내 포인트
- 베팅 입력
- 댓글/좋아요/대댓글

### 10.2 관리자 화면

- 승인 대기 큐
- 이벤트 상세 관리
- 정산 모니터링
- 칭호 규칙 관리

### 10.3 렌더링 원칙

- 공개 읽기 화면은 서버 렌더링 우선
- 로그인 후 상호작용은 클라이언트 섬으로 분리
- 관리자 화면도 같은 Next.js 앱 안의 보호된 경로로 제공
- 주요 사용자 흐름은 PC와 모바일 화면에서 모두 자연스럽게 동작해야 한다.
- 리스트, 이벤트 상세, 리더보드, 관리자 화면은 반응형 레이아웃으로 설계한다.
- 좁은 화면에서는 단일 컬럼 중심, 넓은 화면에서는 정보 밀도를 높인 다중 컬럼 레이아웃을 사용한다.

## 11. 데이터 소스 전략

우선순위:
1. 무료 API
2. 크롤링

설계 원칙:
- 외부 데이터 공급자별 어댑터를 둔다.
- 공급자 포맷은 Source Intake에서 내부 표준 모델로 변환한다.
- 결과 수집 공급자와 오즈 공급자는 분리 가능해야 한다.
- 특정 공급자 장애가 도메인 모델을 오염시키지 않게 한다.

운영 리스크:
- 무료 API는 커버리지와 안정성이 불안정할 수 있다.
- 크롤링은 구조 변경에 취약하다.
- 따라서 공급자 실패는 정상 케이스로 취급하고, 관리자 개입 경로를 기본 제공한다.

## 12. 보안 원칙

- 세션 쿠키는 HttpOnly, Secure, SameSite 정책을 적용한다.
- 관리자 API는 별도 권한 검사를 둔다.
- 댓글, 좋아요, 베팅, 무료 충전, 닉네임 변경은 전부 인증 필요
- 닉네임 변경, 무료 충전, 관리자 액션은 감사 가능한 형태로 기록한다.
- 포인트 원장은 불변 로그처럼 취급하고 직접 수정하지 않는다.

## 13. 테스트 전략

### 13.1 도메인 단위 테스트

- 오즈 컨센서스 계산
- 같은 결과 추가 베팅 규칙
- 반대 결과 변경 금지
- 경기 시작 5분 전 마감 규칙
- 무료 충전 조건
- 정산 계산
- 환불 처리
- 칭호 획득 및 합성

### 13.2 모듈 통합 테스트

- 수집 → 승인 → 공개 → 마감 → 결과 반영 → 정산
- 정산 후 포인트/칭호/리더보드 갱신
- 수동 오즈 입력 후 승인
- 결과 충돌 시 ReviewRequired 전이

### 13.3 외부 연동 계약 테스트

- 무료 API 응답 포맷 변화 감지
- 크롤링 선택자 깨짐 감지
- 공급자 장애 시 안전 정지 확인

### 13.4 핵심 E2E 테스트

- 로그인
- 베팅
- 댓글/좋아요/대댓글
- 무료 충전
- 닉네임 변경
- 관리자 승인
- 자동 정산
- 관리자 수동 재처리

## 14. 미래 확장 방향

- 정치/기타 카테고리는 Category Catalog와 Source Intake 확장으로 수용한다.
- 배당 정책은 현재 고정 odds지만, 장기적으로 사용자 풀 기반 가격 모델로 교체할 수 있게 Odds 정책을 분리한다.
- 관리자 화면은 현재 같은 Next.js 앱 안에 두지만, 필요 시 별도 앱으로 분리 가능하다.

## 15. 구현 시 지켜야 할 핵심 제약

- SourceItem과 PredictionEvent를 혼동하지 않는다.
- PointLedgerEntry 없이 잔액을 변경하지 않는다.
- 정산은 반드시 재실행 안전해야 한다.
- 자동 정산은 데이터 이상 시 멈춰야 한다.
- 칭호와 리더보드는 베팅 코어를 직접 수정하지 않는 후행 계산 구조를 유지한다.
- 프론트는 PC와 모바일을 모두 고려한 반응형 웹으로 설계한다.
